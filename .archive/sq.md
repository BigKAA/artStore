# Изменения в архитектуре приложения

## Пользователи системы

Система не предназначена для непосредственного использования и длогина в нее конечных пользователей.
Системйо будут пользоваться другие приложения, с которыми работают пользователи.
Поэтому аттрибуты пользователей системы будт гораздо проще.
Достаточно указать название клиента и сформировать его UUID, пароль и краткое описание.
Следовательно, пользоватлеи будут только локальные и не надо реализовывать хранение их учетных записей в LDAP и делать OAUTH2 аутентификацию.
Пользователи в базе хранятся все время. Но могут быть помечены как неактивные.

## Файл с атрибутами

Файл *.attr.json, который хранит атрибуты сахроняемого файла должен иметь два типа атрибутов:

1. Постояннные, не изменяемые со временем.
2. Динамичестки изменяемые в различных элемментах хранения.

После того как элемент хранения переводится в режим RO формат файла *.attr.json не должен изменяться.
Но при создании нового элемента хранения, заказчикам может потребоваться добавить или изменить список и название дополнительных атрибутов.

Необходимо продумать как правильно реализовать работу с изменяемыми атрибутами. Варианты которые вижу я: 

1. Неизменяемое поле, содержащее json строку с изменяемыми параметрами.
2. Описывать шаблон изменяемых параметров отдельном файле шаблона, находящемся в корне файловой системы элемента хранения.
